Waystone v0.2.0-rc.1 defines the first release-candidate compatibility surface for the v0.2 migration contract. It adds conformance fixtures and documents the boundaries another tool may consume without guessing.

## Highlights

- Added a v0.2 compatibility policy covering versioned artefacts, source namespaces, migration plans, strategy files, loss reports and archive expectations.
- Added conformance fixtures for single-source, multi-source, local Waystone, migration-plan and loss-report review.
- Hardened ledger durability and archive reading around existing contract surfaces.

## Security

- Trusted operation and source signatures now require the signature public key to match the stored trusted identity public key before Waystone reports the signature as trusted.

## In This Release

- `docs/compatibility.md` now defines the implemented compatibility contract for `waystone.ledger.v1`, `waystone.archive.v1`, `waystone.migration_plan.v1`, `waystone.migration_strategy.v1` and `waystone.migration_loss_report.v1`.
- Compatibility documentation references JSON, timestamp and versioning boundaries using RFC 8259, RFC 3339 and Semantic Versioning.
- `testdata/conformance/` now contains small fixtures for source-scoped identity, cross-source collisions, local labelled issues, safe read-only strategy files, migration-plan verification and structured loss reports.
- Ledger JSON writes now encode before replacing the destination file and use a same-directory temporary file for replacement.
- Archive inspection and import now enforce defensive entry-count and aggregate-size limits in addition to per-entry limits.
- Operation listing now has a deterministic tie-breaker when operations share the same timestamp.
- Migration plan verification now rejects trailing JSON and invalid source namespaces.

## Release Line

`v0.2.0-rc.1` is the first release candidate for the v0.2 bridge-ready migration contract. Later v0.2 prereleases should be limited to blockers, documentation fixes and compatibility corrections.
